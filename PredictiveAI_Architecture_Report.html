1. System Architecture & Services

Three independently deployable services communicate over HTTP/REST and WebSocket.

Simulation Server — Node.js / Express — :3000

Generates deterministic sensor data for 4 machines over a 7-day degradation cycle. Serves /history, /stream (SSE), /alert, /schedule-maintenance. Epoch-anchored seed keeps live and history in sync.

Patterns: bearing_wear, thermal_runaway, cavitation, healthy

Backend API — FastAPI / Python 3.11 — :8000

Core logic: JWT auth, machine management, ML prediction ensemble, alert lifecycle, technician dispatch, work orders, WebSocket broadcast, automation loop (30s), cascade isolation, AI insights.

Frontend — React 18 / Vite / TS — :5173 / :80

15-page SPA. Dark/light theme, WebSocket real-time updates, staged audio alerts (Web Audio API), browser TTS voice alerts, PDF export via jsPDF, command palette (Cmd+K).

PostgreSQL — Railway Managed

Production DB. Auto-created via SQLAlchemy create_all (IF NOT EXISTS). asyncpg driver for non-blocking I/O. URL auto-converted from postgresql:// to postgresql+asyncpg:// by config helper.

2. End-to-End Workflow

1

Sensor Data Generation

Node.js simulation server generates readings for CNC_01, CNC_02, PUMP_03, CONVEYOR_04 using a deterministic 7-day degradation model. Reading = base + linear drift (degrade × progress) + smoothed noise (averaged over 4 adjacent minute-seeds). No sudden jumps — consecutive readings share correlated noise.

2

Backend Automation Loop (every 30 seconds)

  • Fetch latest sensor reading per machine from /history/{id}?limit=1
  • Run 2-step ML verification: anomaly detection (IsoForest + SVM) then supervised (RF + XGBoost)
  • Check failure fingerprint DB (19 patterns, normalised Euclidean distance)
  • Update machine risk_score and failure predictions in DB
  • Check milestone thresholds (72h/48h/24h/12h/6h/1h) — each fires ONCE only
  • Auto-assign best available technician if urgency ≥ high
  • Auto-create maintenance work order if none exists
  • Broadcast WebSocket events to all connected clients
  • Check unacknowledged critical alerts >5 min → auto-escalate
  • Check risk_score >85 → auto-isolate machine, protect downstream
3

Alert Lifecycle

Status flow: active → acknowledged → resolved.
If not acknowledged in 5 minutes: auto-escalated → technician dispatched → work order created → Email + Slack sent → WebSocket broadcast → voice TTS alert.

4

Frontend Real-Time Updates

Dashboard polls /machines and /alerts every 5s via React Query. 4 WebSocket connections (one per machine) receive push events. On pre_failure_alert: toast + staged audio + voice TTS. On cascade_warning: toast + warning audio. On alert_auto_escalated: red toast + critical audio + voice.

5

AI Insights (on demand or auto at phase ≥ 2)

Failure phase classification (0–3), Pearson correlation matrix, ROI calculator, maintenance window optimizer, natural language incident report. Exportable as PDF via jsPDF + html2canvas.

6

Cascade Isolation

Line: CNC_01 → CNC_02 → PUMP_03 → CONVEYOR_04. Any isolated machine cuts flow to all downstream. WebSocket broadcasts machine_isolated event. Operator can manually isolate/release via Cascade Safety page.

4. Machine Learning Algorithms

4.1 Two-Step Verification Pipeline

A machine is only flagged at-risk when BOTH steps agree, minimising false positives.

Step 1 — Unsupervised Anomaly Detection

  • Isolation Forest (sklearn): contamination=0.1, n_estimators=100. Flags readings deviating from normal distribution.
  • One-Class SVM (sklearn): kernel='rbf', nu=0.1. Trained on normal data only — rejects anything outside the learned boundary.
  • If EITHER model flags anomaly → proceed to Step 2.

Step 2 — Supervised Failure Prediction

  • Random Forest Classifier: 100 trees, max_depth=8. Trained on labelled failure CSV data.
  • XGBoost Classifier: n_estimators=100, max_depth=6, learning_rate=0.1.
  • Ensemble: risk = 0.4 × IsoForest_score + 0.3 × SVM_score + 0.3 × avg(RF, XGB)
  • Outputs: risk_score (0-100), hours_to_failure, failure_type, confidence %

Step 3 — Failure Fingerprint Matching (19 patterns)

  • Normalised Euclidean distance: d = sqrt(mean(((current - fp) / fp)^2)) across temperature, vibration, current
  • Match: d < 0.25 (25% avg deviation). Warning: d < 0.40. Time-proximity bonus: ×0.85 if hours_remaining within 6h of fingerprint
  • Fingerprint types: bearing_wear, thermal_runaway, cavitation, seal_failure, belt_slip, roller_wear

6. AI Intelligence Features

Failure Phase Fingerprinting

Classifies each machine into Phase 0-3 based on sensor deviation from known baselines. Estimates Remaining Useful Life (RUL) in hours. Phase ≥2 automatically posts alert to simulation server feed (visible on AI Agent Dashboard).

Sensor Correlation Matrix

Pearson r between temperature, vibration, current, rpm. Identifies cross-sensor anomalies (e.g., temperature rising while RPM drops = bearing seizure risk). Colour-coded heatmap in AI Insights.

ROI Calculator

Compares reactive vs predictive maintenance costs. Annual savings and ROI%. Uses industry-standard rates: $500/h downtime, $8,000 emergency repair, $3,500 unplanned parts.

Maintenance Window Optimizer

Analyses time-of-day load patterns to suggest 3 optimal windows that minimise production impact. Weights off-peak hours (22:00-06:00) highest.

AI Incident Report Generator

Natural language report: phase summary, correlation findings, RUL estimate, recommended actions, predicted failure type. Exportable as PDF via jsPDF + html2canvas.

Voice Alert (Browser TTS)

Critical alerts trigger SpeechSynthesisUtterance (rate=1.1, pitch=0.9). Example: "Critical alert. CNC Machine 1 predicted to fail in 2 hours." No-ops gracefully if TTS unavailable.

5-Minute Auto-Escalation

If a critical alert is not acknowledged within 5 minutes: autonomous technician dispatch, priority work order creation, Email + Slack notifications, WebSocket broadcast, critical audio + voice alert.

Automated Sim-Server Reporting

When failure phase ≥ 2: backend POSTs structured alert to simulation server /alert endpoint. Visible as AI Agent activity on the Agent Dashboard page in real time.

7. Cascade Failure Prevention

Production line: CNC_01 (Zone A) → CNC_02 (Zone A) → PUMP_03 (Zone B) → CONVEYOR_04 (Zone C)

Topology Rules

  • CNC_01 isolated → CNC_02, PUMP_03, CONVEYOR_04 all blocked (full line halted)
  • CNC_02 isolated → PUMP_03, CONVEYOR_04 blocked
  • PUMP_03 isolated → CONVEYOR_04 blocked
  • CONVEYOR_04 is terminal — no downstream dependencies

Auto-Isolation (risk_score > 85)

  • Automation loop checks risk_score each 30s cycle
  • risk_score > 85 → MachineIsolation record created (isolation_type='auto')
  • Machine status set to 'isolated' in DB
  • All downstream machines show "Flow Blocked" status in UI
  • WebSocket broadcasts machine_isolated event with protectedMachines list
  • Alert created: title = "Machine auto-isolated due to critical risk"

Pre-Isolation Cascade Warning

  • If risk jumps >15 in one cycle AND downstream machine risk >30 → cascade_warning broadcast
  • Frontend shows toast + warning audio before any isolation occurs

Manual Isolation (Operator)

  • POST /api/v1/isolation/isolate {machineId, reason} — requires operator role
  • Same DB record + WebSocket broadcast as auto-isolation
  • POST /api/v1/isolation/release to restore machine after repair

10. Production Issues Fixed

ModuleNotFoundError: app.models.isolation Critical
Problem: main.py imported isolation router which imported MachineIsolation from a non-existent file. Module was scaffolded in import statements but never created.
Fix: Created backend/app/models/isolation.py (MachineIsolation SQLAlchemy model with full schema: machine_id, is_isolated, isolation_type, reason, risk_at_isolation, protected_machines). Also created app/services/fingerprint_service.py with match_fingerprints() normalised Euclidean distance function and 19 pre-seeded fingerprint patterns.
CORS: credentials + wildcard origin conflict Critical
Problem: FastAPI CORSMiddleware raises error when allow_origins=['*'] is combined with allow_credentials=True (CORS spec: wildcard cannot be used with credentials). Railway frontend was being blocked by browser CORS policy.
Fix: Added _wildcard check in main.py: allow_credentials = not ('*' in origins). When CORS_ORIGINS='["*"]', credentials mode is disabled so browsers accept the wildcard.
Auth loading hang — blank white screen High
Problem: Axios had no timeout configured. If the Railway backend was slow to respond, the /auth/me request hung indefinitely, keeping isLoading=true forever and showing a blank screen.
Fix: Added timeout: 10000 (10s) to apiClient axios instance. Added 401/403 response interceptor to clear localStorage token and redirect to /login.
Railway frontend: API URL hardcoded to localhost:8000 High
Problem: Vite bakes VITE_API_URL at build time inside the Docker image. The Railway frontend always made requests to localhost:8000 instead of the Railway backend URL.
Fix: Runtime injection — nginx Dockerfile CMD generates config.js from Railway env vars API_URL and WS_URL before starting nginx. apiClient.ts reads window.__API_URL__ at runtime. Same image works everywhere.
Sudden data spikes in sensor charts High
Problem: (1) Live readings used Math.floor(Date.now()/1000) as random seed — new random value every second caused values to jump independently. (2) Live and history used different 7-day cycle anchors (UNIX epoch mod vs rolling window from now), creating discontinuities when plotted together. (3) CNC_02 thermal anomaly was a hard step change the instant hourOfDay crossed 13.00.
Fix: (1+2) Both live and history now use same minute-based seed i*37+1 anchored to UNIX epoch cycle start (Date.now() % CYCLE_MS). (3) CNC_02 thermal spike ramps smoothly over 30 minutes via min(rampIn, rampOut) envelope. (4) Noise averaged over 4 adjacent seeds for autocorrelated drift.
uvicorn --reload hangs on background task shutdown
Problem: start.sh launched uvicorn with --reload. WatchFiles detected changes in agent/config.py (outside backend dir), triggering graceful shutdown. The automation asyncio task was stuck in asyncio.sleep(30) and would not cancel, causing the server to hang indefinitely on "Waiting for background tasks to complete."
Fix: Removed --reload from start.sh. Background daemons should not auto-reload; --reload is only for interactive development.
36-machine cascade topology vs 4-machine reality
Problem: Cascade isolation was expanded to M01-M36 (36 machines) but only 4 machines exist in the simulation server and database.
Fix: Simplified MACHINE_TOPOLOGY to the 4 real machines (CNC_01, CNC_02, PUMP_03, CONVEYOR_04) with correct upstream/downstream relationships. CascadeIsolationPage rebuilt as a clean 4-machine production line flow diagram.
